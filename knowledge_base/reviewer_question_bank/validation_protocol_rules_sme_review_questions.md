# SME Review Questions: Validation Protocol Rules

## Purpose

This question bank supports SME / QA / RA / method owner review of the validation protocol rule set.

The goal is to help reviewers identify whether the rules are suitable for internal draft generation and where revisions are needed before broader use.

---

## 1. Overall Rule Suitability

1. Are the rules suitable as first-pass internal draft guidance?
2. Are there any sections that could be misread as final approval logic?
3. Are the limitations of AI-assisted generation clear enough?
4. Are human review gates consistently preserved?
5. Are evidence levels and confidence labels sufficiently clear?

---

## 2. Product and Method Scope

1. Are the rules too specific to recombinant protein drug product release/stability?
2. Are any product-specific assumptions hidden in the generic rule language?
3. Are small molecule, peptide, antibody, oligonucleotide, vaccine, or cell/gene therapy extensions blocked by the current design?
4. Which product classes should require separate rule extensions?
5. Which method types require SME-specific review before use?

---

## 3. Validation Characteristic Selection

1. Are the selected validation characteristics reasonable for each method type?
2. Are any required characteristics missing for assay, potency, impurity, or identity methods?
3. Are any characteristics over-included where they may not apply?
4. Are bioassay and immunoassay considerations such as matrix effect and parallelism handled appropriately?
5. Are microbial and endotoxin methods handled in a way that avoids inappropriate forced mapping?

---

## 4. Validation Scope Logic

1. Is the distinction between full validation, partial validation, method transfer, co-validation, and verification clear?
2. Are triggers for partial validation described at a useful level?
3. Are transfer responsibilities and comparative criteria sufficiently addressed?
4. Are verification rules cautious enough to avoid under-validation?
5. What additional scope categories should be added later?

---

## 5. Acceptance Criteria Logic

1. Does the rule set avoid inventing unsupported numerical limits?
2. Are criteria rationale fields sufficient?
3. Is the link between acceptance criteria, method purpose, and reportable result clear?
4. Does the rule set require development data or prior evidence where needed?
5. Are AI-inferred criteria clearly identified as needing review?

---

## 6. System Suitability and Deviation Handling

1. Are system suitability expectations linked to critical method performance?
2. Are system suitability examples appropriate and not misleading?
3. Are invalid run, failed run, repeat, retest, and data exclusion expectations clear?
4. Is QA involvement clear where deviation handling may affect conclusions?
5. Are any deviation rules too prescriptive for early draft use?

---

## 7. Report Review and Feedback Loop

1. Can protocol planning gaps be converted into report review questions?
2. Are the protocol-to-gap mapping rules useful for reviewer triage?
3. Are feedback targets clear enough to improve the knowledge base over time?
4. Are gaps ranked with useful severity categories?
5. Should additional gap IDs be added before broader use?

---

## 8. Reviewer Decision Questions

1. Can this rule set be used for internal draft generation after documented review?
2. What revisions are required before use?
3. Are any sections unsuitable for use without additional SME input?
4. Should any method-specific rule card be created next?
5. Should this review generate new backlog items, rule updates, or template updates?
